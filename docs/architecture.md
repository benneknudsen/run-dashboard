## 2. Database Schema (Drizzle + Vercel Postgres)

- **users** вЂ” linked to Strava athlete
- **strava_tokens** вЂ” AES-256-GCM encrypted OAuth tokens (GDPR)
- **activities** вЂ” running workouts from Strava with full metrics
- **ai_analyses** вЂ” cached AI responses with `inputHash` deduplication
- **chat_messages** вЂ” Phase 2 RAG conversations
- **activity_embeddings** вЂ” Phase 2 pgvector with HNSW index

### Key security decisions:
- OAuth tokens encrypted at rest (AES-256-GCM, per-row IV)
- `inputHash` deduplication вЂ” AI calls once per dataset, not per page view
- Raw Strava payload stored for audit/re-processing

## 5. AI Integration Architecture

### Provider Routing
Abstraction layer that selects primary model, falls back to secondary on failure. One swap point for all AI calls.

### Generative UI Flow (streamUI)
1. Server loads user's activities + computed metrics
2. `streamUI({ model, system, prompt, tools })` with Zod-validated tool definitions
3. Model interleaves streamed prose with tool-driven component renders
4. Mixed UI tree streams into `<analysis-panel>` with `<Suspense>` boundary

### Streaming Patterns
- `useObject` for structured analysis with known shape
- `streamUI` + tools for flexible generative UI
- `useChat` for Phase 2 RAG chatbot
- Independent `<Suspense>` boundaries: stats, charts, AI panel never block each other

### RAG Architecture (Phase 2)
1. On sync: generate NL summary в†’ embed в†’ store in pgvector (HNSW, cosine)
2. On query: embed в†’ vector search (scoped to userId) в†’ inject into system prompt
3. k=4-6 chunks, always filtered by userId (prevents cross-user leakage)

## 7. State Management (Non-overlapping)

| Layer | Tool | Holds |
|-------|------|-------|
| Server cache | TanStack Query | Activities, refetch, sync status |
| AI streaming | Vercel AI SDK | useObject, useChat streams |
| UI | Zustand | Theme, filters, selected activity |

> Server data never in Zustand. AI state never in Query. Clean separation is the senior signal.

## 8. Performance Strategy

- **PPR**: Static shell prerendered, dynamic content streams via Suspense
- **Suspense boundaries**: Independent boundaries for stats, charts, AI panel
- **Edge webhook**: Low-latency 200 response, heavy work deferred to node
- **Tag-based revalidation**: Event-driven (on sync), not time-based ISR
- **AI cost control**: `inputHash` caching вЂ” model called once per dataset

## 9. Phase 1 MVP Scope

### Build:
- Strava OAuth + encrypted token storage + refresh
- Activity sync (backfill 30 activities)
- Dashboard: stats, weekly volume chart, pace distribution, activity list
- Activity detail: map, splits, HR zones
- **One** AI flow: streaming generative-UI weekly insight (streamUI + 3-4 primitives)
- Polished empty/loading/error states
- **Demo mode**: seeded fixture data (hiring managers won't connect Strava)
- Deployed to Vercel

### Deliberately excluded:

- Predictive ML
- Training plan generation

## 10. Key Architectural Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| ORM | Drizzle over Prisma | SQL-first, edge-compatible, faster cold starts, shows SQL fluency |
| Auth | NextAuth v5 over Clerk | Free, no vendor lock-in, demonstrates OAuth understanding |
| AI location | Server-side only | Keys never reach browser, controllable caching, GDPR boundaries |
| UI paradigm | Generative UI over plain text | Typed components, model on rails, 2026 AI-first signal |
| Caching | Event-driven over ISR | Running data changes on new activity only, not on timer |
